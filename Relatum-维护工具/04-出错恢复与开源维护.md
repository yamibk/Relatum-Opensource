# 出错恢复与开源维护

## 情况一：AI 改错了，但还没有 commit

先运行：

```cmd
git status --short
git diff --stat
```

然后告诉 AI：

```text
先只读检查 git status 和 git diff，区分本轮 AI 修改与已有修改。
不要使用 git reset --hard、git clean、git restore . 或任何强制操作。
只用精确补丁撤销你本轮造成的代码变化，不要碰 data、canvases 和我的其他修改。
```

不要因为一个文件出错就删除整个项目文件夹。

## 情况二：已经 commit，但还没 push

把以下输出交给 AI：

```cmd
git status -sb
git log --oneline -5
```

让 AI 判断是追加修复 commit，还是安全撤销刚才的 commit。不要自行执行 `reset --hard`。

## 情况三：错误已经 push 到 GitHub

最安全的做法是“向前修复”：

1. 修好问题；
2. 创建一个新的 commit；
3. 再次 push。

如果需要完整撤销某次公开提交，让 AI 使用 `git revert` 生成一个反向 commit。不要删除云端仓库，也不要 `push --force` 改写公开历史。

## 情况四：GitHub 上被网页或贡献者改过

本地工作区干净时运行：

```cmd
git pull --ff-only origin main
```

如果命令提示冲突、分叉或 non-fast-forward，停止操作，把完整输出交给 AI。不要用强推覆盖对方修改。

## 情况五：误上传 API Key

处理顺序必须是：

1. 立即到服务商后台吊销或轮换密钥；
2. 确认新密钥没有写入仓库；
3. 再让 AI 判断是否需要清理 Git 历史；
4. 检查 Release、Issue、日志和截图里是否也出现过。

只删除当前文件不能让旧提交里的密钥失效，因此第一步永远是撤销密钥。

## 情况六：Release 上传错了

- 附件尚未正式发布：在草稿中修正。
- Release 已经公开：不要替换同一版本号下的附件；标明旧版本问题，修复后发布新的补丁版本，例如从 `vX.Y.0` 升到 `vX.Y.1`。

不要把不同源码构建出的程序悄悄放在同一个版本号下面。

## 什么时候才需要分支

你现在单人维护、小步修改时，直接使用 `main` 足够。

出现以下情况时，再让 AI 创建分支：

- 一个大功能可能做几天并且可能放弃；
- 多个 AI 或多人同时修改；
- 想先让别人审查再进入正式版本；
- 接受外部 Pull Request。

这时只需告诉 AI：

```text
这个任务风险较大，请基于最新 main 创建独立分支。
修改、测试和提交都留在分支，不要直接推送或合并 main。
完成后向我解释差异和风险，再决定是否合并。
```

不需要现在学习复杂的 merge 或 rebase 命令。

## 外部贡献者的最小规则

- 让贡献者通过 fork 和 Pull Request 提交，不直接给予陌生人仓库写权限。
- 一个 Pull Request 尽量只解决一个问题。
- 让 AI 阅读 PR、检查用户数据、许可证、性能风险和测试结果。
- GitHub Actions 通过不代表交互一定正确，仍需人工打开相关页面测试。
- 新增图片、字体、音频和第三方代码必须说明来源与许可证。
- 不要求问题报告者公开上传真实画布、`data`、API Key 或私人截图。
- API Key 泄露或安全漏洞不要发布为普通公开 Issue；应使用 GitHub 的私密安全报告渠道。普通 Issue 只使用完全虚构的最小复现数据。

## 永远保留的安全底线

- 不删除 `.git`。
- 不强推 `main`。
- 不重写已公开的版本标签。
- 不把 `Relatum-release` 当源码。
- 不在源码仓库里保存真实用户数据。
- 不让 AI 在有未确认修改时整棵回滚。
- 不清空 GitHub 仓库后重新上传。
- 每次先检查、再 commit；先测试、再 Release。
