# 让 AI 修改代码后的日常流程

## 一、只在 Git 仓库开发

以后所有 AI 都应该只修改真正的 Relatum Git 仓库。仓库可以位于任意磁盘，但根目录必须包含：

```text
.git\
app.py
assets\
build-desktop.ps1
```

不要把以下位置当成新源码继续修改：

- 以前的历史版本工程；
- `Relatum-release` 或解压后的安装包；
- 从 GitHub 点“Download ZIP”得到的文件夹；
- 个人日常使用的 Relatum 数据目录。

建议把“个人使用副本”“源码开发测试”和“待上传的构建输出”分开：

- 个人使用副本是你从 Release ZIP 另外解压出来、用于日常工作的那一份，可以产生真实数据；
- 开发仓库只用空白或虚构测试数据；
- `Relatum-release` 构建输出是待检查、压缩和上传的暂存目录，绝不能用来保存真实数据；
- AI 不得修改 `data/`、`canvases/` 或真实 `.canvas`。

## 二、每次任务开始前

在仓库根目录打开 CMD，然后运行：

```cmd
git status -sb
```

如果输出只有这一行，下面没有任何文件记录：

```text
## main...origin/main
```

才说明工作区干净。同样的第一行下面如果还列出了文件，工作区仍然有修改。

如果列出了修改文件，先把输出交给 AI 判断这些修改是谁留下的，不要直接覆盖或拉取。

工作区干净，并且 GitHub 可能被网页编辑或其他贡献者更新过时，再运行：

```cmd
git pull --ff-only origin main
```

## 三、给 AI 的推荐开场指令

可以直接复制：

```text
只在当前这个 Relatum Git 仓库工作；仓库根目录包含 .git、app.py 和 assets。

开始前先读取 AGENTS.md 和 CONTRIBUTING.md，并检查 git status。
不要修改 data、canvases、真实 .canvas、历史工程和 Relatum-release。
只修改本任务需要的文件，完成后运行相关语法检查、功能测试和 scripts/check-public.ps1。
先用通俗语言告诉我改了哪些文件、测试结果和风险；未经我明确要求，不要 commit、push、强制回滚或打包 Release。
```

这个方式仍然允许 AI 直接改源码，只是增加了边界和验收。

## 四、AI 改完以后

先让 AI 完成：

1. 解释修改内容；
2. 列出实际改过的文件；
3. 运行与修改有关的测试；
4. 检查是否误碰用户数据或隐私；
5. 检查 `AGENTS.md` 是否需要同步更新。

你也可以在 CMD 中查看概要：

```cmd
git status --short
git diff --stat
powershell -NoProfile -ExecutionPolicy Bypass -File ".\scripts\check-public.ps1"
```

日常推送使用默认安全扫描即可。`-Physical` 会连被忽略的本地测试数据也检查，主要用于制作公开 ZIP 或干净副本；如果它发现 `data` 或 `canvases`，不要在未备份的情况下删除这些目录。

## 五、确认正确后保存并同步

### 推荐方式：继续让 AI 处理

复制下面的指令：

```text
请复核本轮所有 git diff，确认只有本任务修改。
重新运行必要测试和 scripts/check-public.ps1。
检查通过后，把本轮修改创建为一个说明清楚的 commit，并推送到 origin/main。
不要使用 --force，不要上传 data、canvases、API Key、缓存或打包产物。
完成后告诉我 commit 编号、推送结果和 GitHub Actions 结果。
```

### 仓库维护者的手工备用方式

```cmd
git add -A
git status --short
git diff --cached --stat
git diff --cached --name-status
powershell -NoProfile -ExecutionPolicy Bypass -File ".\scripts\check-public.ps1"
git commit -m "fix: describe this change"
git push origin main
```

`git add -A` 之后一定要再次看 `git status --short`。如果清单里出现不认识的私人文件，先停止，不要 commit。

这组命令只适合仓库维护者。外部贡献者应当在自己的 fork/分支中提交并创建 Pull Request，不直接推送项目的 `main`。网络或代理报错时，把完整输出交给 AI，不要照抄其他电脑的代理地址。

## 六、什么时候打包

不是每次 push 都需要打包。

- 小修复、文档更新、开发中的功能：测试后 commit + push 即可。
- 准备给用户使用的稳定版本：源码 push 成功后，再执行打包和 GitHub Release 流程。

## 七、最简日常清单

- [ ] 我确认 AI 工作在正确的本地仓库。
- [ ] 开始前查看了 `git status`。
- [ ] AI 没有碰用户数据和历史工程。
- [ ] AI 解释了修改并运行测试。
- [ ] 安全扫描通过。
- [ ] 正确修改已经 commit。
- [ ] commit 已 push 到 `origin/main`。
- [ ] GitHub Actions 是绿色。
- [ ] 只有稳定版本才继续打包 Release。
